Audit model

Server audit is canonical. Regex is evidence gathering.

A cached audit must never trust user-machine code. The client sends package identity and optional integrity; the server resolves from npm, downloads the npm tarball, verifies registry integrity, extracts bounded evidence, and uses the user's Anthropic/OpenAI key only for cache misses.

Target-aware request

The CLI requests an audit for a target such as npx. Future npm-install audits use the same pipeline, but a different prompt, cache key, and risk policy.

Current state

The foundation exists. Now make the trust flow sharp.

We have enough code to dogfood the concept. The next work is not a public site; it is making betternpx/bnpx reliable as npx replacements, while betternpm/bnpm stay npm-compatible until install auditing is ready.

Core package

Shared inspector engine. Parses specs, resolves npm metadata, queries downloads and OSV, scans tarballs locally, scores risk, and caches exact package inspections.

CLI package

Command surface for both families: betternpx/bnpx replace npx with inspect-before-exec, while betternpm/bnpm replace npm. Ordinary npm commands pass through; direct package installs are inspected before npm runs.

Server audit

Cloudflare Worker resolves from npm, verifies integrity, scans npm tarballs, supports target-aware D1 cache keys, and has BYOK provider plumbing.

Verified commands

npm run check, package inspection, local Worker audit creation/cache hits, summary lookup, and exact allow-cache execution all pass locally.

Action points

The product roadmap is the trust layer.

P0

Make betternpx/bnpx excellent

  • betternpx create-next-app and bnpx create-next-app should work as safer replacements for npx create-next-app.
  • betternpm and bnpm should behave like npm first, inspect direct package installs now, then add full dependency graph inspection as soon as that path is trustworthy.
  • Resolve the exact npm version, audit it, and run only that exact artifact.
  • Make approve, block, --force-install, --force-fresh-audit, JSON, and non-interactive behavior boringly reliable.
P1

Make server audit canonical

  • Never accept client-submitted source for cacheable audits.
  • Always pull metadata and tarballs from npm on the server.
  • Reject caller integrity values that do not match npm registry metadata.
P2

Make BYOK audit the main security layer

  • Use anthropic or openai from CLI config.
  • Send the user key only for server-side cache misses or explicit forced refreshes.
  • Store normalized risk output, model, provider, audit date, and future authenticated user ID only: no provider key, no raw user secrets.
  • Server high/blocked audit results must block execution unless the user passes --force-install.
P3

Reduce regex to evidence gathering

  • Use deterministic scans to find files and snippets worth reviewing.
  • Feed source evidence into the BYOK prompt.
  • Show source links for manual review whenever npm metadata gives us repository and commit data.
P4

Stabilize the report contract

  • Keep a stable shape for level, score, summary, findings, evidence, sourceUrl, and cache identity.
  • Use the same contract for CLI, API, future package pages, and future npm-install audits.
  • Make reports specific enough that a developer can decide without guessing.
P5

Build the package test set

  • Known good: create-next-app, vite, tsx, typescript, shadcn.
  • Legitimate weird: bundled CLIs, child processes, env reads, generated files.
  • Suspicious: new packages, low downloads, install scripts, obfuscated bundles, public malicious advisories where usable.
P6

Publish the CLI when the trust flow feels right

  • First public product is not the website. It is: run betternpx/bnpx instead of npx, and betternpm/bnpm instead of npm.
  • Publish @betternpm/core and @betternpm/cli, exposing betternpm, bnpm, betternpx, and bnpx bins.
  • Reserve betternpx as the public package name. betternpm and bnpm are already taken, and npm rejected unscoped bnpx as too similar to existing packages, but all three can still be CLI bin aliases from @betternpm/cli.
  • Add a GitHub-source installer path so curl https://betternpm.org/latest | sh can install/link the CLI before npm publishing is fully polished.
P7

Deploy the same trust layer

  • Deploy apps/api to Cloudflare Workers with D1 migrations applied.
  • Deploy the public site to Vercel only after external copy is ready, pointing browser audit calls at api.betternpm.org.
  • Serve the installer script from betternpm.org/latest as a proxy to the GitHub script or release artifact.
P8

Research npm install only after npx works

  • Resolve and audit dependency graphs.
  • Inspect install scripts and transitive package risk.
  • Respect lockfiles and start as a wrapper before pretending to be a package manager replacement.

Security boundaries

The promise is visibility before execution, not magic sandboxing.

Never execute inspected tarballs

The inspector downloads packages for analysis only. Execution remains delegated to npm after approval.

Cache by content

Audit records key on target, package name, version, integrity, scanner profile, provider, and model. Name/version alone is not enough.

User keys are request-only

BYOK means the Worker receives a provider key only for cache misses or forced refreshes and stores normalized audit output, model, date, and future user attribution.

Explicit overrides

High or blocked packages require --force-install. Experts can still proceed, but casual execution gets friction.

Known limits

The MVP does not replace npm installs, revoke packages, or fully sandbox executed code.

Scanner profiles

Every deterministic scoring version is named, so caches can be invalidated when the rubric changes.

The product promise stays narrow: safer npx. Better npm audits the exact npm artifact on the server before you run it.
